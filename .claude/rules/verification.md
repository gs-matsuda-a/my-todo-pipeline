# 検証（テスト通過≠問題解決）

## 原則

ユニットテストが緑でも「問題が解決した」ことにはならない。テストは実装と同じ誤解を共有しうる
（同じ作者が同じ思い込みでテストと実装を書くため）。**「完了」を宣言できるのは、動いている実物で
要求どおりの振る舞いを実証できたときだけ**。

## 実装後スモーク（PR 前の必須ゲート）

- PR を作る前に、アプリを**実際に起動**し、主要フローを HTTP（または実際の入出力経路）で
  1 往復して確認する。Web アプリの例: 登録 → ログイン → 主要リソースの作成 → 取得 → 更新 → 削除。
- **起動しない・主要フローが通らない状態で PR を作ってはならない**（テストが緑でも）。
- 確認したコマンドと結果を PR 本文のテストプランに記載する。

## 契約テスト先行（外部仕様書があるとき）

- API 仕様書（例: `api-spec.md`）が入力に与えられた場合、**最初に仕様書から契約テストを生成**する。
  エンドポイント・リクエスト/レスポンスの形式（フィールド名・ネスト・型）・ステータスコードを
  仕様書の記載どおりに検証するテスト。
- 以降のすべての Issue・リファクタは契約テストを green に保つ。
  **テストの「正解」はモデルの理解ではなく仕様書に固定する**（自分の理解でテストを書くと、
  実装と同じ誤解を検証してしまう）。
- 複数 Issue に分解する場合、契約テストの整備を最初の Issue の受け入れ条件に含める。

## 統合の検証（Issue 分割開発の縫い目）

- 各 Issue の完了時に「その Issue のテスト」だけでなく**全テスト＋起動スモーク**を実行する。
- 分割の縫い目（ルーティングの優先順位・認証ミドルウェアの適用範囲・データ形式の食い違い・
  ライブラリ設定）は単体テストに出ない。スモークだけが検出できる。
