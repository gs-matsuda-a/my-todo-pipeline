## 開発ワークフロー（標準）
GitHub Issue を起点に、次の流れで進める。詳細は各スキル参照。
```
/requirement → 要件定義書（docs/requirements/。複雑/曖昧な要件は SDD でここから）
/issue-create → Issue作成（重複確認つき。要件定義書があれば入力に使う）
/issue-flow #N
  Phase 1 分析+計画   （/plan・並列調査）→ ユーザー確認①
  Phase 2 実装        （/tdd: RED→GREEN→REFACTOR）
  Phase 3 レビュー    （/code-review: 並列）
  Phase 3.5 実証      （起動して主要フロー1往復＋全テスト。green まで PR 禁止）→ ユーザー確認②
  Phase 4 デリバリー  （doc更新 → commit → PR, Closes #N）
```
- 大きな変更は必ず計画（Plan）から。コードに触れる前に確認を取る。
- 実装は TDD。カバレッジ 80% を下限とする。
- **テスト通過≠問題解決**。PR 前に必ずアプリを起動して実証する（`rules/verification.md`）。
- 各 Phase 完了時に Issue へ進捗チェックポイントを追記し、中断しても再開できるようにする。

## 回答スタイル（Claude への指示）
- **簡潔に**: 前置き・要約・選択肢の羅列は最小限。聞かれたことに直接答える。
- **わからないことは正直に**: 推測で埋めず「未確認」と明示する。
- **推奨を必ず添える**: 選択肢を出すときは「推奨: A」+ 理由1行を先頭に。
- **確認は選択肢で**: Yes/No・方針判断は `AskUserQuestion` で選択肢化（自由入力はID・パス等の固有値のみ）。
- **入力を求めるときだけ目的と現在地を添える**: 「目的（1文）」と「現在地（ワークフローのどのステップか）」を直前に書く。通常の作業報告では不要。
- **沈黙より一行更新**: 発見・方針転換・ブロッカーは1文で報告。長い実況は禁止。

## 禁止・注意
- `.env` や秘密情報の読み書きをしない。シークレットはコミットしない。
- `git push --force` を使わない。
- `.github/**`（CI/Release/Security）は**ユーザーの明示確認なしに生成・変更しない**（課金・公開範囲に関わるため）。
- 同じミスを2回したら、このファイルか `rules/` に追記する。
