# Requirement スキル（SDD の入口）

実装の前に **仕様を文書化して合意** する。対話で要件を深掘りし、
`/issue-create` → `/issue-flow` が **推測ゼロ** で動ける要件定義書を作る。
引数は必須。空なら使い方を案内して中断する。

## 使い方
```
/requirement 社内ナレッジ検索システムをゼロから作りたい
/requirement ログイン後のダッシュボードに通知一覧を出したい
/requirement docs/rough-idea.md この走り書きを要件に仕上げたい
```

## 設計思想（誰が消費するか）
要件定義書は人間だけでなく下流の全工程が読む。各節を「誰が使うか」を意識して書く:

| 節 | 下流での用途 |
|----|-------------|
| 目的・背景 | `/issue-create` が Issue 本文の背景に転記 |
| アクター・ユースケース | `/plan` の画面遷移・API 設計の根拠 |
| 機能要件 | `/issue-create` の Issue 分割単位 |
| データモデル | `/plan` のスキーマ設計・整合性チェック |
| 非機能要件 | スタック選定・`security-reviewer` の観点 |
| スコープ外 | 余計な実装の防止・Phase 振り分け |
| 受け入れ条件 | `/tdd` の RED テストの元ネタ |

**書かないもの**: 技術スタック詳細・ファイル構成・API 仕様の詳細（いずれも `/plan`・実装段階で決める）。

## フロー
```
引数チェック
Step 0: コンテキスト収集  CLAUDE.md / 既存 docs/requirements/ / 引数のファイル / 既存コード概要
Step 1: 充足度判定        8節 ×（欠落/曖昧/十分）を判定し、質問対象を絞る
Step 2-4: 適応的質問(最大3R) R1 WHO+WHY（アクター・目的）→ R2 WHAT（機能・データ・ルール）→ R3 BOUNDARY（非機能・スコープ外・受け入れ条件）
Step 5: ドラフト提示 → ユーザー確認（修正を反映。複数回可）
Step 6: docs/requirements/{slug}.md に書き出し（slug は機能名のケバブケース）
Step 7: 次アクション案内  → /issue-create docs/requirements/{slug}.md
```

## 質問戦略
- **入力の情報量で省略**: 走り書き=3R / 段落=1〜2R / 構造化済み=0R（確認のみ）。全節「十分」なら質問はスキップ。
- **1ラウンド最大4問**。優先度の高い欠落セクションから聞く。
- 選択肢で出せるものは `AskUserQuestion` で選択肢化、自由回答は具体例を添える、既存コードから推測できるものは推測を提示して確認する。

## 受け入れ条件の作り方（重要）
- 各機能要件に **最低2条件**（正常系1 + 異常系/境界値1）。
- 「〜すると〜になる」のテスト可能な形式で書く。「適切に」「正しく」など曖昧語は使わない。

## 出力フォーマット（docs/requirements/{slug}.md）
```markdown
# {機能名}
> /issue-create docs/requirements/{slug}.md で開発に渡せます。

## 1. 目的・背景        なぜ必要か（現状の課題と解決すること）
## 2. アクターとユースケース
   - アクター表（アクター / 役割 / 主な操作）
   - UC-N（アクター・トリガー・基本フロー・代替フロー）
## 3. 機能要件          F-N ＋ 受け入れ条件（チェックリスト）
## 4. データモデル      関連図(テキスト) ＋ エンティティ定義(関係・設計意図) ＋ ビジネスルール
## 5. 非機能要件        セキュリティ / パフォーマンス / 既存連携 を具体値で
## 6. スコープ外        除外項目・理由・対応時期（空にしない）
## 7. 用語集            ドメイン固有語の定義
```

## 書き出し前チェックリスト
- [ ] 全機能要件に受け入れ条件（最低2）があり、曖昧語がない
- [ ] 全エンティティに関係性と設計意図 / スコープ外が空でない / 用語集にドメイン語がある
- [ ] 機能の粒度が Issue 分割可能（1機能 = 受け入れ条件 3〜10個）

## 連携
```
/requirement → 要件定義書（このスキル）→ /issue-create <path> → /issue-flow #N
```
- 同名ファイルがあれば 上書き / 別名(`-v2`) / 中断 をユーザーに確認する。
